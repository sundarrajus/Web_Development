Modules (import / export)

Why Modules?

Before modules, all JS files shared one global scope — variables in one file could accidentally overwrite variables in another. Modules solve this by giving each file its own private scope. You explicitly export what others can use, and explicitly import what you need.

Typical Folder Structure

project/
  index.html
  main.js             ← entry point
  utils/
    math.js          ← utility module
    strings.js
  models/
    User.js          ← class module

Named Exports

utils/math.js
// Named export — export multiple things from one file
export const PI = 3.14159;

export function add(a, b)      { return a + b; }
export function subtract(a, b) { return a - b; }
export function multiply(a, b) { return a * b; }

// Can also export at the bottom in a list
// export { PI, add, subtract, multiply };
main.js
// Named import — must use the exact exported names
import { PI, add, multiply } from './utils/math.js';

console.log(PI);           // 3.14159
console.log(add(3, 4));    // 7
console.log(multiply(3, 4)); // 12

// Rename on import with 'as'
import { add as sum } from './utils/math.js';
console.log(sum(10, 20));  // 30

Default Export

Each file can have only one default export. It is usually the main thing the file is about — a class, a function, or a component.
models/User.js
// Default export — one per file
export default class User {
    constructor(name, email) {
        this.name  = name;
        this.email = email;
    }
    greet() { return `Hello, ${this.name}`; }
}
main.js
// Default import — you choose ANY name (no curly braces)
import User from './models/User.js';

const u = new User("Sundar", "sundar@example.com");
console.log(u.greet());   // Hello, Sundar

Import all as namespace (*)

// Import everything from a module under one namespace object
import * as MathUtils from './utils/math.js';

console.log(MathUtils.PI);           // 3.14159
console.log(MathUtils.add(5, 3));    // 8

Re-exporting (barrel files)

Pattern: Create an index.js inside a folder that re-exports everything. Consumers then import from the folder, not individual files.
utils/index.js
export { add, subtract, multiply, PI } from './math.js';
export { capitalize, trim }           from './strings.js';
main.js
// Clean single import from the folder
import { add, capitalize } from './utils';

Dynamic import() — lazy loading

// Loads the module only when needed (returns a Promise)
async function loadHeavyFeature() {
    const { heavyFunction } = await import('./heavy-module.js');
    heavyFunction();
}

// Useful for code-splitting — don't load code the user may never need
document.getElementById("btn").addEventListener("click", async () => {
    const module = await import('./chart.js');
    module.renderChart(data);
});

Browser Modules vs Node.js

// BROWSER — use type="module" in HTML; file paths need extensions
// <script type="module" src="main.js"></script>

// NODE.js (CommonJS — older style)
const math = require('./math');         // import
module.exports = { add, subtract };     // export

// NODE.js (ES Modules — modern, add "type": "module" in package.json)
import { add } from './math.js';
export { add };
Note: This file demonstrates module concepts with code examples. Since modules require a server or bundler (like Vite or Webpack) to work in the browser, the interactive demo is not runnable as a plain HTML file. When you build a real project, use <script type="module"> and a dev server.
